# 001 全体計画: PresentsAI 製品化

## アーキテクチャ方針

「最初からモノリスにしない」を絶対原則とする。
Canva / Google Slides スケールを見越し、各サービスが独立してスケールできる構造を初日から確立する。

## バックエンド アーキテクチャ詳細（Hexagonal Architecture）

`services/api` は六角形アーキテクチャ（Ports and Adapters）で実装する。

```
┌─────────────────────────────────────────────┐
│              domain 層                       │
│  - エンティティ定義（User, Presentation等）  │
│  - リポジトリインターフェース（interface）   │
│  - ドメインサービス                          │
│  外部依存: ゼロ（標準ライブラリのみ）        │
└────────────────────┬────────────────────────┘
                     │ 依存方向（上から下）
┌────────────────────▼────────────────────────┐
│              application 層                  │
│  - ユースケース（CreatePresentation等）      │
│  - domain のインターフェース経由で操作       │
│  外部依存: domain のみ                       │
└────────────────────┬────────────────────────┘
                     │
┌────────────────────▼────────────────────────┐
│            infrastructure 層                 │
│  - postgres/: GORM リポジトリ実装            │
│  - storage/: ローカル/S3 実装               │
│  - http/: Gorilla Mux ハンドラー            │
│  外部依存: PostgreSQL, ファイルシステム等    │
└─────────────────────────────────────────────┘
```

**メリット:**
- ビジネスロジックが HTTP / DB に依存しない
- テストが容易（リポジトリをモックするだけ）
- 将来 DB を変えても domain / application 層は無変更
- サービス分割時に domain + application をそのまま移植可能

## フロントエンド アーキテクチャ詳細（Feature Slices）

`web/src/features/` 配下は「縦断スライス」構造。

```
features/editor/
├── components/
│   ├── Canvas/           ← Fabric.js キャンバス本体
│   │   ├── Canvas.tsx    ← メインコンポーネント
│   │   ├── useCanvas.ts  ← Fabric.js 操作フック
│   │   └── index.ts
│   ├── Toolbar/          ← ツール選択バー（上部）
│   ├── PropertyPanel/    ← 右側プロパティパネル
│   ├── SlidePanel/       ← 左側スライドサムネイル
│   └── MenuBar/          ← ファイル/編集/表示メニュー
├── stores/
│   ├── editorStore.ts    ← 現在のツール、選択状態
│   ├── slideStore.ts     ← スライドデータ（Yjs連携）
│   └── historyStore.ts   ← Undo/Redo
├── hooks/
│   ├── useCanvas.ts
│   ├── useKeyboard.ts    ← ショートカットキー
│   └── useCollaboration.ts ← Yjs接続
└── api.ts                ← この機能が使うAPI呼び出し（他の機能から直接呼ばれない）
```

各 feature は**他の feature を直接 import しない**。共有が必要なものは `shared/` に昇格させる。
これにより将来 feature を別パッケージ/マイクロフロントエンドに分割可能。

## 並行エージェント戦略

- 同一フェーズ内の独立した PR は **worktree isolation** で並行実行
- backend PR と frontend PR は同時着手可能（インターフェース先行合意後）
- インターフェース（API 仕様）を先に確定し、フロントとバックを並行実装
